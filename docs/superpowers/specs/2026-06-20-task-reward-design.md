# LifeBoard 待办激励模块设计

日期：2026-06-20

## 目标

在 `LifeBoard` 内新增一个待办激励模块，用于管理长期项目、维护任务模板、生成当天任务，并在任务完成后累积“可支配奖励额度”。

第一版目标不是做通用项目管理器，也不是做工时系统，更不是把行为激励直接写入真实账本。它只解决一个最小闭环：

1. 从长期项目中维护任务模板
2. 手动挑选当天要做的任务
3. 完成任务后获得奖励额度
4. 手动扣减奖励额度并记录原因

## 范围

### 包含

- `项目 -> 任务模板 -> 当天任务` 两层半结构
- 任务模板默认预计时长
- 任务模板默认奖励金额
- 手动将模板任务加入某一天
- 完成任务时发放奖励
- 手动扣减奖励余额
- 奖励余额、今日已赚、完成进度、预计总时长展示
- 奖励流水历史查看

### 不包含

- 计时器
- 番茄钟
- 自动提醒
- 自动结转到次日
- 严格惩罚机制
- 真实财务交易自动联动
- 奖励扣减正式分类体系
- 独立历史分析页
- 日历视图

## 产品原则

- 模块属于 `LifeBoard`，但业务域独立于现有财务账本
- 激励额度是“行为奖励”，不是“真实资金”
- 今日页是主战场，项目页是配置区，奖励页是流水区
- 第一版以低摩擦使用为优先，不增加不必要验证和状态
- 默认使用正向激励，不做失败惩罚

## 核心业务模型

### Project

长期事项容器，例如“健身”“写作”“学习英语”。

建议字段：

- `id`
- `name`
- `status`：`active` / `archived`
- `sort_order`
- `created_at`
- `updated_at`

### TaskTemplate

项目下的模板任务，例如“跑步 30 分钟”“写 800 字”。

建议字段：

- `id`
- `project_id`
- `name`
- `default_estimated_duration_minutes`
- `default_reward_amount`
- `notes`
- `is_active`
- `created_at`
- `updated_at`

说明：

- 默认奖励金额和默认预计时长定义在模板层
- 加入今日列表时允许覆盖

### DailyTask

某一天实际执行的任务实例。它从模板创建，但必须保存快照，避免模板修改影响历史记录。

建议字段：

- `id`
- `date`
- `project_id`
- `task_template_id`
- `name_snapshot`
- `estimated_duration_minutes_snapshot`
- `reward_amount_snapshot`
- `status`：`pending` / `completed`
- `actual_duration_minutes`
- `completed_at`
- `created_at`
- `updated_at`

说明：

- 第一版不引入 `in_progress`
- 第一版不引入 `skipped`
- 第一版不引入自动结转标记

### RewardLedger

激励额度流水，只记录奖励系统内部的增减，不进入真实财务流水。

建议字段：

- `id`
- `entry_type`：`earn` / `spend` / `adjust`
- `amount`
- `reason`
- `daily_task_id`
- `created_at`

说明：

- 第一版主要使用 `earn` 与 `spend`
- `adjust` 预留给未来人工校正
- 当前奖励余额不单独存表字段，通过流水汇总计算

## 金额与时间存储规则

- 奖励金额使用“分”存储，前端按“元”展示
- 时长单位统一使用“分钟”
- 日期使用自然日概念，围绕本地日期组织 `DailyTask`

## 页面信息架构

模块作为 `LifeBoard` 内新导航分组，新增三个视图：

1. `今日`
2. `项目`
3. `奖励`

### 今日页

这是第一优先页面，也是模块主入口。

页面应包含：

- 当前奖励余额
- 今日已赚金额
- 今日完成进度
- 今日预计总时长
- 今日任务列表
- 从模板加入今日任务的入口
- 手动扣减奖励的入口

今日任务列表每项显示：

- 任务名称
- 所属项目
- 预计时长
- 奖励金额
- 完成状态

每项任务支持：

- 修改任务名称
- 修改预计时长
- 修改奖励金额
- 标记完成

### 项目页

这是配置和维护区域，不是主工作台。

页面应包含：

- 项目列表
- 当前项目下的任务模板列表
- 新增项目
- 新增模板任务
- 编辑模板任务
- 停用模板任务

模板任务显示：

- 模板名称
- 默认预计时长
- 默认奖励金额
- 备注
- 是否启用

### 奖励页

这是激励额度的流水查看与扣减区域。

页面应包含：

- 当前余额
- 累计赚取
- 累计扣减
- 奖励流水列表
- 手动扣减入口

流水项显示：

- 增减方向
- 金额
- 原因
- 关联任务
- 时间

## 核心交互

### 从模板加入今日

用户在今日页或项目页选择模板任务，将其加入指定日期。默认带出：

- 模板名称
- 默认预计时长
- 默认奖励金额

用户可在创建时覆盖预计时长和奖励金额。

### 完成任务

用户点击完成时：

1. 可选填写实际时长
2. 提交完成请求
3. 后端将任务状态更新为 `completed`
4. 后端自动新增一条 `RewardLedger(earn)`
5. 前端刷新任务状态与奖励摘要

完成不需要强制填写总结、备注或证据。

### 扣减奖励

用户在奖励页或今日页点击“手动扣减奖励”时：

1. 输入金额
2. 输入原因文本
3. 提交扣减请求
4. 后端新增一条 `RewardLedger(spend)`
5. 前端刷新余额与流水

第一版原因使用自由文本，不做分类体系。

### 未完成任务

未完成任务不会自动结转到第二天。若次日还想做，应重新加入今日列表。

## API 设计

后端沿用当前仓库风格：`api` 层保持薄路由，主要业务逻辑进入 `services`。

### 项目与模板

- `GET /api/task-projects`
- `POST /api/task-projects`
- `PATCH /api/task-projects/{id}`
- `GET /api/task-templates?project_id=...`
- `POST /api/task-templates`
- `PATCH /api/task-templates/{id}`

### 当天任务

- `GET /api/daily-tasks?date=YYYY-MM-DD`
- `POST /api/daily-tasks`
- `PATCH /api/daily-tasks/{id}`
- `POST /api/daily-tasks/{id}/complete`
- `POST /api/daily-tasks/{id}/reopen`

说明：

- `POST /api/daily-tasks` 用于从模板加入某一天，也允许创建时覆盖时长和奖励
- `complete` 请求体仅包含可选的 `actual_duration_minutes`
- `reopen` 第一版建议保留，用于纠正误操作

### 奖励

- `GET /api/rewards/summary`
- `GET /api/rewards/ledger?limit=...`
- `POST /api/rewards/spend`

## 状态流转

### DailyTask

- `pending -> completed`
- `completed -> pending` 通过 `reopen`

不引入其他状态，避免第一版出现无价值复杂度。

### RewardLedger

- 完成任务时生成 `earn`
- 手动扣减时生成 `spend`
- 未来需要人工修正时可生成 `adjust`

## 关键约束

### 完成接口必须幂等

`complete` 接口必须防止重复发奖励。如果任务已完成，再次请求应返回已完成结果，而不是继续生成奖励流水。

### 重开任务必须冲销奖励

如果允许 `reopen`，则必须同步冲销此前因该任务产生的奖励额度，不能只改任务状态，否则余额会错误。

实现方式可以是：

- 新增一条负向 `adjust` 流水

或：

- 将原有 `earn` 标记失效

第一版推荐使用“新增冲销流水”，因为审计轨迹更清楚。

### 奖励扣减不能透支

`POST /api/rewards/spend` 必须校验当前余额，余额不足时拒绝请求并返回明确错误。

### 模板停用不影响历史

模板一旦停用：

- 不允许再加入今日
- 已存在的 `DailyTask` 历史记录保持不变

## 前端交互细节

### 今日页卡片

顶部建议使用 4 个核心摘要卡片：

- 当前奖励余额
- 今日已赚
- 完成进度
- 预计总时长

奖励额度应作为第一优先视觉指标。

### 完成弹层

完成任务时使用轻量确认层，只包含：

- 任务标题确认
- 可选的实际时长输入

不要增加多余字段。

### 扣减弹层

扣减弹层只包含：

- 金额输入
- 原因文本输入

不要做复杂分类、标签或消费映射。

## 错误处理

- 重复完成任务：提示“该任务已完成”
- 奖励余额不足：提示“余额不足”
- 模板已停用：提示“该模板已停用，无法加入今日”
- 记录保存失败：保留用户输入并提示重试

## 测试策略

### 后端测试

至少覆盖以下场景：

- 完成任务后生成奖励流水
- 重复完成不重复发奖励
- 重开任务时奖励被冲销
- 扣减奖励时余额校验生效
- 模板停用后不能再创建新的 `DailyTask`

### 前端测试

第一版至少覆盖关键流程：

- 今日列表加载成功
- 完成任务弹层提交流程
- 完成后摘要卡片刷新
- 扣减奖励成功与余额不足提示

## 与现有系统的关系

- 该模块属于 `LifeBoard` 产品内部
- 与现有财务仪表盘共享整体应用壳、路由和视觉体系
- 不复用现有 `transactions` 表
- 不把奖励额度写成真实财务记录

后续若需要联动真实预算或账户，应在奖励系统稳定后单独设计映射策略，而不是在第一版直接混用数据域。

## 未来演进方向

以下能力明确留到后续版本：

- 周报或月报统计
- 日历视图
- 自动生成今日任务建议
- 提醒与通知
- 计时器
- 奖励额度与真实预算联动
- 项目级奖励统计
- 更细的奖励原因分类

## 结论

第一版应实现一个边界清晰的待办激励模块：

- 在项目下维护模板任务
- 手动加入当天任务
- 完成后获得奖励额度
- 手动扣减奖励额度
- 以今日页作为主战场

这能在不污染现有财务模型的前提下，建立一个可持续迭代的激励闭环。
